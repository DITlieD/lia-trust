# LIA Trust Kernel v0.2.1

Patch release over `v0.2.0`. Fixes false PreToolUse denials when Grok Build reuses the
Claude-compatible global hook path.

## Highlights

- **Grok hook envelope compatibility** in the `claude-code` adapter:
  - Accept camelCase / alias top-level fields: `toolName`, `toolInput`, `hookEventName`,
    `sessionId`, `toolUseId`, `permissionMode`, `workspaceRoot`, `transcriptPath` (plus snake_case).
  - Normalize Grok tool names onto Claude gates before dispatch:
    `run_terminal_command`/`shell`/`bash` → `Bash`, `read_file`/`read` → `Read`,
    `search_replace`/`StrReplace`/`edit` → `Edit`, `write`/`WriteFile` → `Write`,
    `multi_edit` → `MultiEdit`, `notebook_edit` → `NotebookEdit`,
    `delete_file`/`delete` → `Delete`.
  - Accept path aliases on gated tools: `file_path` | `target_file` | `path` | `filePath` |
    `targetFile`.
  - Accept write/edit body aliases: `content` | `new_string` | `contents` | `newString` | `text`.
  - Normalize `pre_tool_use` / `preToolUse` → `PreToolUse`.
- Fail-closed is unchanged when the tool name is truly missing, and filesystem-scope still
  denies paths outside `allowed_roots`.
- Unmapped tools still fail-open with reason `"no gate mapped for tool"` — intentional for
  Claude-native tools not in the seven-core map. Grok names that previously fell into that
  path (or failed parse entirely) are now mapped so gates actually run.

## Why this shipped

Grok loads `~/.claude/settings.json` PreToolUse hooks when `compat.claude.hooks` is enabled.
It sends a camelCase envelope (`toolName` / `toolInput` / `target_file`) while the adapter
required Claude snake_case (`tool_name` / `tool_input` / `file_path`). Parse failure exited 2
(fail-closed deny) before any gate evaluated — a false-deny storm on `read_file`,
`run_terminal_command`, and `search_replace`.

## Honest limitations

Same mediation boundaries as `v0.2.0`: hooks only gate observed calls; complete mediation is not
claimed. Unmapped Claude tool names remain fail-open by design.
